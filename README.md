## ü§ù Contribution Guidelines

We welcome contributions from the community to make `ticket_konser` even better!

### Code Style

*   **HTML:** Use semantic HTML5 elements. Ensure proper indentation and closing tags.
*   **CSS:** Follow a consistent naming convention (e.g., BEM). Organize styles logically.
*   **JavaScript:** Adhere to modern JavaScript best practices. Use JSDoc comments for complex functions.

### Branch Naming Conventions

Please use the following prefixes for your branches:

*   `feature/` for new features (e.g., `feature/user-login`)
*   `bugfix/` for bug fixes (e.g., `bugfix/responsive-layout`)
*   `hotfix/` for urgent fixes (e.g., `hotfix/critical-error`)
*   `docs/` for documentation updates (e.g., `docs/update-readme`)

### Pull Request Process

1.  **Open an Issue:** Before starting work, please open an issue to discuss your proposed changes or features.
2.  **Fork the Repository:** Fork the `ticket_konser` repository to your GitHub account.
3.  **Create a New Branch:** Create a new branch from `main` with an appropriate name (see above).
4.  **Make Your Changes:** Implement your feature or fix, ensuring all code adheres to the style guidelines.
5.  **Commit Your Changes:** Write clear, concise commit messages.
6.  **Push to Your Fork:** Push your new branch to your forked repository.
7.  **Submit a Pull Request:** Open a pull request against the `main` branch of the original repository. Provide a detailed description of your changes.

### Testing Requirements

*   **Manual Testing:** Thoroughly test your changes in different browsers and screen sizes.
*   **Code Review:** Be prepared for your code to be reviewed by maintainers.
